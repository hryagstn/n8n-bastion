# Case Study: PaketLebaranku.id

This document describes a real-world deployment of n8n-bastion on a production
Laravel application. It serves as a reference implementation for the setup guide.

---

## Background

**PaketLebaranku.id** is a seasonal e-commerce platform built with Laravel,
running on an Ubuntu VPS with Apache as the web server.

In May 2026, the server experienced a filesystem-level intrusion. Obfuscated PHP
backdoors were planted in non-standard directories (e.g. `public/icons/`), and
`.htaccess` files were modified to allow execution of Python scripts through Apache.
The first indication was the disappearance of the `.env` file, followed by a
persistent HTTP 500 error even after it was restored.

The investigation revealed:
- Multiple `index.php` files containing `eval(base64_decode(...))` patterns
- A modified `.htaccess` with `AddHandler application/x-httpd-python .php`
- The entry point was never conclusively identified (likely a file upload vulnerability)

This incident directly motivated two open-source tools:

- **[laravel-scalpel](https://github.com/hryagstn/laravel-scalpel)** — a filesystem intrusion evidence scanner for Laravel
- **[n8n-bastion](https://github.com/hryagstn/n8n-bastion)** — this repository, for automated monitoring and alerting

## Active Watchdogs

All four n8n-bastion watchdogs are running in production:

| Watchdog | Status | Notes |
|----------|--------|-------|
| Filesystem Sentinel | ✅ Active | Scans every 5 min via laravel-scalpel |
| Apache Health Monitor | ✅ Active | Checks every 1 min, auto-restarts if needed |
| Resource Monitor | ✅ Active | Disk 85% / RAM 90% threshold |
| HTTP Uptime Check | ✅ Active | Double-confirms before alerting |

## Lessons Learned

**Always create the baseline after a clean deployment.**
Running `scalpel:baseline` before `vendor:publish` or `composer install` will
cause false positives on subsequent `scalpel:diff` runs.

**The double-check in uptime-check.sh matters.**
Transient network issues caused one false alarm before the 30-second confirmation
delay was implemented. The delay eliminated false positives entirely.

**n8n Execute Command is not available in n8n 2.0+ by default.**
The initial design used Execute Command nodes. After hitting this limitation,
the architecture was reworked to Bash scripts + Webhook, which is lighter and
more reliable anyway.

**bootstrap/cache should NOT be excluded from the baseline.**
Files like `services.php` and `packages.php` are generated by Laravel — but
they are also high-value targets for attackers who want to inject malicious
service providers. Monitor them, and re-baseline after `php artisan optimize`.

---

## Maintenance Workflow

After every deployment to PaketLebaranku.id:

```bash
cd /path/to/paketlebaranku
php artisan optimize
php artisan scalpel:baseline --force
```

This keeps the baseline in sync and prevents false positives from
legitimate framework-generated file changes.
